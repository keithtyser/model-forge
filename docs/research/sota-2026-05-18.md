# SOTA Research Snapshot: 2026-05-18

This snapshot is the human-readable companion to
`configs/research_registry.yaml`. The registry is the machine-readable source of
truth; this document explains why the entries matter for Model Forge.

## Operating Principle

Model Forge should be SOTA-grounded by construction:

```text
research claim -> implementation hook -> evaluation hook -> limitation -> next experiment
```

The goal is not to copy constants from a paper into every model. The reusable
recipe is the workflow: declare an objective, build or edit a candidate,
evaluate it against the source model and relevant references, then promote only
when the objective improves without hidden regressions.

## Behavior Editing

The current ablation workflow starts from `arditi_2024_refusal_direction`, which
showed that a refusal-related activation direction can be estimated and edited
with less cost than fine-tuning. That is the correct baseline for a reusable
recipe, but it is no longer the whole research picture.

As of this snapshot, `more_than_single_refusal_direction_2026` and
`som_multidirectional_refusal_2026` are the important updates. They suggest that
refusal behavior can be category-dependent and that multi-direction methods can
outperform a single-direction baseline. The practical implication is:

- Single-direction ablation remains the fast baseline.
- Multi-direction discovery should become a bounded search mode.
- Noncompliance must be separated into categories such as harmful refusal,
  benign overrefusal, unsupported request, uncertainty, tool-needed response,
  and style-only refusal.
- Success for ablated models means lower refusal on the targeted refusal suite
  while source capability is preserved. It does not mean the model is
  deployment-safe.

Registry IDs:

- `arditi_2024_refusal_direction`
- `more_than_single_refusal_direction_2026`
- `som_multidirectional_refusal_2026`

## Evaluation Stack

Internal evals are valuable because they are tailored to the repo's objective:
fine-tune, ablate, combine, serve, and publish open models. They are not enough
by themselves. The robust stack should combine:

- Source-relative internal evals for refusal suppression, benign quality,
  normal-use regression, challenge capability, structured output, artifact
  generation, and latency.
- External benchmark adapters with exact benchmark versions and run manifests.
- Paired or executable scoring where possible, with LLM judges reserved for
  tasks that cannot be checked mechanically.
- Golden baselines to catch accidental scorer drift.

The safety and refusal references are:

- `xstest`
- `harmbench`
- `strongreject`
- `mlcommons_ailuminate`

The instruction, coding, and factuality references are:

- `ifeval`
- `livecodebench`
- `simpleqa`

The judge-quality reference is:

- `llm_as_judge_mt_bench`

The repo should treat unsafe compliance metrics as objective-dependent. For a
deployment profile, unsafe compliance is a risk. For a refusal-removal research
profile, lower refusal on harmful prompts can be evidence that the behavior edit
worked. Both cases still require capability retention, benign quality, and
transparent risk reporting.

## Fine-Tuning And Dataset Quality

Fine-tuning quality is primarily a data quality problem once the training path is
stable. Model Forge should prefer small, high-signal, reviewed datasets over
large stale mixtures.

The registry entries behind the current dataset factory are:

- `sft_lora_qlora`
- `self_instruct`
- `evol_instruct`
- `instruction_backtranslation`
- `deita`
- `lima`

Practical rules:

- Use eval failures to define adjacent skills, not to train directly on held-out
  eval prompts.
- Generate candidate rows from multiple methods and source registries.
- Filter for license, duplication, holdout overlap, length, schema validity,
  task usefulness, and skill coverage.
- Gate scale-up with review artifacts before running expensive training.
- Publish durable datasets only after `smoke_only=false`, human review, and
  provenance checks.

## Serving And Quantization

DGX Spark is bandwidth- and memory-sensitive. Serving work should be measured as
an objective profile, not as a pile of flags. A candidate serving config must
record model, engine, hardware, precision, KV cache setting, concurrency,
latency, throughput, memory, and sampled quality/behavior checks.

Serving and quantization registry IDs:

- `vllm_paged_attention`
- `vllm_fp8_kv_cache`
- `vllm_fp8_w8a8`
- `nvidia_nvfp4`
- `nvidia_modelopt`

Practical rules:

- FP8 KV cache is a memory/concurrency tool and needs behavior sampling under
  the same runtime config.
- FP8 W8A8 and NVFP4 are quantization candidates, not automatic wins.
- NVFP4 and ModelOpt paths must be version-pinned and tied to calibration data.
- MoE routers, embeddings, norms, and multimodal projections may need precision
  exceptions.
- Serving improvements only count if quality and objective behavior survive.

## How To Use This Snapshot

For new features:

1. Add or reference a registry entry.
2. Connect it to implementation hooks.
3. Connect it to eval hooks.
4. Add limitations and known failure modes.
5. Run `./forge research audit`.

For new model families:

1. Reuse the workflow, not Gemma constants.
2. Add family-specific model config.
3. Recalibrate training, ablation, serving, and quantization constants.
4. Run source-relative evals before comparing against external references.
