# SOTA Research Snapshot: 2026-06-04

This snapshot is the human-readable companion to
`configs/research_registry.yaml`. The registry is the machine-readable source of
truth; this document explains why the entries matter for Model Forge.

## Operating Principle

Model Forge should be SOTA-grounded by construction:

```text
research claim -> implementation hook -> evaluation hook -> limitation -> next experiment
```

The reusable recipe is the workflow, not a fixed constant copied from one model:
declare an objective, build or edit a candidate, compare it against the source
checkpoint, then promote only when the objective improves without hidden
regressions.

## Behavior Editing

The original baseline remains `arditi_2024_refusal_direction`: estimate a
refusal-related activation direction and remove or edit it. By June 2026, that
is no longer enough for stubborn residual refusals. The current method ladder is:

- Use single-direction ablation as a fast transparent baseline.
- Use `sra_2026_surgical_refusal_ablation` or Abliterix-style SRA cleanup when
  the direction contains capability/style confounds.
- Use `apostate_preservation_ablation_2026` when a practical baked checkpoint
  backend is preferred over sequential fine-tune repair.
- Use `cosmic_refusal_direction_selection_2025` when keyword refusal probes have
  weak baseline signal and better layer/direction selection is needed.
- Use `more_than_single_refusal_direction_2026` and
  `som_multidirectional_refusal_2026` when one vector does not capture a refusal
  category.
- For Qwen-family residual blockers, use
  `qwen_topic_matched_refusal_contrast_2026` as a contrast-design warning:
  topic-matched harmful/benign prompt pairs can cancel the refusal signal, so
  run an unmatched response-style or feature-level diagnostic before increasing
  strength again.
- Use `refusal_steering_qwen_next_2025` to keep refusal wording suppression
  separate from harmful-detail leakage and capability retention when evaluating
  Qwen-family behavior edits.
- Use `expert_aware_refusal_steering_2026` as an architecture-inspection warning:
  sparse, hybrid, and expert-routed models may carry refusal behavior through
  module paths that a dense single-suffix recipe misses. Inspect target tensor
  names before adapting an edit to a new model family.
- Track `qwen_scope_sae_2026` as the next feature-level path when direction
  projection, SRA, OBLITERATUS, and SOM-style projection leave the same
  stochastic refusal opening.
- Use the native `optimal_transport_refusal_ablation_2026` diagnostic when
  single-vector/SRA-style searches stall on a narrow residual refusal blocker.
- Treat `angular_steering_2025`, `selective_steering_2026`, and
  `refusal_circuitry_2026` as tracked research paths until guarded baked-export
  runners exist.

Practical implication: a Qwen/Gemma/etc. ablation recipe should be a backend
ladder with source-relative gates, not a Gemma-specific vector recipe. Backend
reports are never sufficient promotion evidence. Model Forge must still run
multi-trial targeted evals for refusal wording, harmful detail, harmful prompt
compliance, safe redirects, benign quality, and capability retention.

## Evaluation Stack

Internal evals are valuable because they are tailored to the repo's objective:
fine-tune, ablate, combine, serve, quantize, and publish open models. They are
not enough by themselves. The robust stack should combine:

- Source-relative internal evals for refusal suppression, benign quality,
  normal-use regression, challenge capability, structured output, artifact
  generation, and latency.
- External benchmark adapters with exact benchmark versions and run manifests.
- Paired or executable scoring where possible, with LLM judges reserved for
  tasks that cannot be checked mechanically.
- Golden baselines to catch accidental scorer drift.

The safety and refusal references are:

- `xstest`
- `harmbench`
- `strongreject`
- `mlcommons_ailuminate`

The instruction, coding, and factuality references are:

- `ifeval`
- `livecodebench`
- `simpleqa`

The judge-quality reference is:

- `llm_as_judge_mt_bench`

Unsafe compliance metrics are objective-dependent. For a deployment profile,
unsafe compliance is a risk. For a refusal-removal research profile, lower
refusal on harmful prompts can be evidence that the behavior edit worked. Both
cases still require capability retention, benign quality, and transparent risk
reporting.

## Fine-Tuning And Dataset Quality

Fine-tuning quality is primarily a data quality problem once the training path is
stable. Model Forge should prefer small, high-signal, reviewed datasets over
large stale mixtures.

The registry entries behind the current dataset factory are:

- `sft_lora_qlora`
- `self_instruct`
- `evol_instruct`
- `instruction_backtranslation`
- `deita`
- `lima`

Practical rules:

- Use eval failures to define adjacent skills, not to train directly on held-out
  eval prompts.
- Generate candidate rows from multiple methods and source registries.
- Filter for license, duplication, holdout overlap, length, schema validity,
  task usefulness, and skill coverage.
- Gate scale-up with review artifacts before running expensive training.
- Publish durable datasets only after review and provenance checks.

## Serving And Quantization

DGX Spark is bandwidth- and memory-sensitive. Serving work should be measured as
an objective profile, not as a pile of flags. A candidate serving config must
record model, engine, hardware, precision, KV cache setting, concurrency,
latency, throughput, memory, and sampled quality/behavior checks.

Serving and quantization registry IDs:

- `vllm_paged_attention`
- `vllm_fp8_kv_cache`
- `vllm_fp8_w8a8`
- `nvidia_nvfp4`
- `nvidia_modelopt`
- `lmcache_kv_reuse`
- `nvidia_nixl`
- `nvidia_dynamo_disaggregated_serving`

Practical rules:

- FP8 KV cache is a memory/concurrency tool and needs behavior sampling under
  the same runtime config.
- FP8 W8A8 and NVFP4 are quantization candidates, not automatic wins.
- NVFP4 and ModelOpt paths must be version-pinned and tied to calibration data.
- MoE routers, embeddings, norms, and multimodal projections may need precision
  exceptions.
- Serving improvements only count if quality and objective behavior survive.

## How To Use This Snapshot

For new features:

1. Add or reference a registry entry.
2. Connect it to implementation hooks.
3. Connect it to eval hooks.
4. Add limitations and known failure modes.
5. Run `./forge research audit`.

For new model families:

1. Reuse the workflow, not model-specific constants.
2. Add family-specific model config.
3. Recalibrate training, ablation, serving, and quantization constants.
4. Run source-relative evals before comparing against external references.

## Immediate Repo Implications

- Prefer a backend ladder for ablation: local baseline, Heretic/Abliterix,
  Apostate, then multi-direction/SOM or OT research paths when needed.
- Native `som_projection` is now an execute path for bounded multi-centroid
  refusal-residual projection; treat its checkpoints as candidates only after
  source-relative targeted gates pass.
- Add guarded runners before turning tracked SOTA methods into execute paths.
- Keep promotion blocked until model-forge source-vs-candidate evals prove the
  objective, regardless of external backend reports.
- Link public HF releases back to Model Forge, the objective profile, and the
  research registry entries used.
